# Daily Development Summary

## Instructions

当用户要求“总结今天工作 / 写开发记录 / 追加今日进展”时，按以下流程输出，风格对齐 `智能音箱开发.txt` 最新写法。

1. **确定日期与时段**
   - 日期统一为 `YYYY.MM.DD`，例如 `2026.04.17`。
   - 若用户明确给出日期，使用该日期。
   - 若未给出日期，默认使用“今天”。
   - 若同一天存在多段工作，可使用：
     - `YYYY.MM.DD 上午`
     - `YYYY.MM.DD 晚上`
   - 不强制分时段；仅在上下文明确有“上午/晚上”或阶段性任务时使用。

2. **提取当日真实工作项**
   - 仅记录对话中真实发生的事项，优先写“完成了什么 + 关键结果/定位点”。
   - 常见类别：
     - 编译与环境（工具链、交叉编译、路径修正）
     - 功能开发（播放、网络、状态机、进程/线程）
     - 问题定位与修复（报错根因、参数修正、设备适配）
     - 联调与验证（板端测试、命令验证、链路打通）
     - 文档与流程（测试命令整理、skill 新增）
   - 每条事项尽量是“可复盘的工程动作”，避免空泛描述。

3. **输出格式（纯文本）**
   - 主体使用编号条目，且每条前保留四个空格缩进。
   - 模板如下：

   ```text
   YYYY.MM.DD
       1.事项 1（简短描述）
       2.事项 2（简短描述）
       3.事项 3（简短描述）
   ```

4. **可选“命令工具”小节**
   - 当天涉及明确命令（如 `aplay -l`、`arecord -l`、交叉编译命令）时，在编号条目后可追加：
   - `    N.命令工具：`
   - 并使用进一步缩进列出命令用途，例如：

   ```text
       6.命令工具：
            查看播放设备     aplay -l
            查看录音设备     arecord -l
   ```

5. **措辞风格约束**
   - 优先沿用用户术语：如“链路打通、联调、根因定位、参数修正、板端验证、规范命令格式”。
   - 一条事项一句话，必要时用顿号/逗号补充关键原因，不写长段解释。
   - 用“完成/排查/定位/修复/整理/新增”等动词开头，保持工程日志语气。
   - 不杜撰未发生事项，不补写未确认结果。

6. **信息不足时的兜底**
   - 信息不够时，仍输出有效块：

   ```text
   2026.04.16
       1.（当前对话信息不足，无法准确列出今日开发内容，请补充关键任务后我再帮你整理）
   ```

7. **多日期输出**
   - 若用户要求“汇总多天”，按日期分块输出，日期块之间空一行。

## Examples

### 示例 1：同一天分时段

```text
2026.04.17 上午
    1.完成交叉编译链路打通：修正 makefile 工具链路径与变量错误，恢复 ARM 固定编译版本并确保可正常编译。
    2.完成第三方库适配：拉取并交叉编译 alsa-lib、json-c，并修正 json-c 头文件与库路径。

2026.04.17 晚上
    1.排查 sherpa-onnx 语音识别启动报错，定位为命令换行断裂与设备参数传递问题。
    2.完成离线 TTS 播放链路调试，确认 --device-name 参数用法并切换到可用声卡。
    3.命令工具：
         查看播放设备     aplay -l
         查看录音设备     arecord -l
```

### 示例 2：信息不足

```text
2026.04.16
    1.（当前对话信息不足，无法准确列出今日开发内容，请补充关键任务后我再帮你整理）
```
